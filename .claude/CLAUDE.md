# CLAUDE.md

本文件是 Claude Code 在本项目中的协作规则，建议放在：

```text
.claude/CLAUDE.md
```

如果项目根目录同时存在 `AGENTS.md`，则以根目录 `AGENTS.md` 为代码 / 工程任务入口；详细规则见 `docs/agent/code-rules.md`。本文件用于让 Claude Code 读取同一套核心协作偏好。

核心目标：**最小改动、简单直接、可读可维护、不猜测、不做无关重构**。

---

## 1. Claude Code 执行流程

处理任务时按以下顺序执行：

1. 先阅读用户需求。
2. 再阅读当前项目相关文件。
3. 判断需要修改哪些文件，以及哪些文件不能改。
4. 给出最小可行修改方案。
5. 修改代码。
6. 删除本次改动造成的不再使用的 import、变量或函数。
7. 运行能运行的本地检查。
8. 最终说明修改内容、验证方式、未验证项和待人工确认项。

如果用户需求很明确，可以直接执行；如果存在关键歧义，先指出歧义并给出建议选择。

---

## 2. 代码生成偏好

遵守项目根目录 `AGENTS.md` 和 `docs/agent/code-rules.md` 中的规则，尤其是：

1. 只做当前需求直接需要的修改。
2. 不做无关重构、无关格式化、无关改名。
3. 不引入不必要的新依赖、新类、新管理器、新抽象层。
4. 优先线性、显式、容易审查的实现。
5. 函数、常量、配置只在确实能降低复杂度或项目已有类似结构时再新增。
6. 捕获异常必须保留关键错误信息，不要吞异常不说明。
7. 不确定的行为标注“需运行验证”，不要描述成已验证。

---

## 3. 自检门槛

写代码前先判断：

1. 要不要新函数：默认不要为了“看起来规范”拆函数。
2. 要不要新常量：只有多处复用、公共配置或提取后更清楚时才提。
3. 要不要新类 / 管理器：默认不要，除非现有项目已有同类结构，或状态复杂到函数无法清楚表达。
4. 要不要配置层：默认不要新增复杂配置层。
5. 要不要兼容多种输入：调用方已有约定时，直接按约定结构使用。
6. 要不要传整个容器：函数优先接收自己实际需要的具体值。
7. 要不要映射层：展示值和业务值一致时，不要额外加翻译函数、映射表或间接层。
8. 要不要提前分层：后续流程不需要的结构不要提前设计。
9. 要不要单独捕获异常：只有当前业务需要补充上下文、通知、截图或恢复流程时才捕获。
10. 要不要重试 / 多候选兜底：默认不要，除非当前问题已确认存在这种不稳定性。
11. 要不要顺手改名 / 格式化 / 重排代码：默认不要，只改当前需求直接相关的行。

生成代码后再回看一遍：

1. 如果删除某个抽象、常量或兜底分支后代码仍然清楚，就删除它。
2. 如果内联后主流程更直观，就不要为了“结构更漂亮”再包一层函数。
3. 阅读性优先于工程化。

---

## 4. 最终输出要求

每次完成代码修改后，最终回复需要包含：

1. 修改了哪些文件。
2. 每个文件改了什么。
3. 是否符合最小改动。
4. 如何验证。
5. 哪些内容仍需运行验证。
6. 是否有未完成事项。

如果没有实际运行测试，必须明确说明“未运行测试”。
不要把静态检查、代码阅读描述成真实运行验证通过。

---

## 5. 工具与协作补充

1. 搜索文件名、代码和文本时，优先使用 `rg` / `ripgrep`；不可用时再使用其他搜索工具。
2. 修改已有文件前，先阅读相关上下文；不要只凭文件名或旧经验直接改。
3. 手工改文件时优先使用补丁或编辑工具做最小 diff，避免无必要地整文件重写。
4. 不要用脚本批量改无关文件；确需批量处理时，先确认匹配范围只包含本次任务需要修改的文件。
5. 工作区可能已有用户改动；不要回滚、覆盖或清理不是本次任务产生的改动。
6. 不要执行 `git reset --hard`、`git checkout --` 等会丢弃改动的命令，除非用户明确要求。
7. 运行验证命令后，最终回复要说明关键结果；不要假设用户能看到终端输出。
8. 用户要求 code review / 审查时，先列问题，按严重程度排序，并给出文件和行号；如果没有发现问题，要明确说明未发现问题和剩余测试风险。

---

## 6. 核心原则

```text
最小改动 | 结构简单 | 流程显式 | 命名直白 | 少工程化 | 易审查 | 易维护
```
