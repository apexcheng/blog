这个博客的核心选择很朴素：先把内容、页面和发布链路做稳，再考虑更复杂的后台能力。

它不是一个完整 CMS，也不是动态应用。文章是仓库里的 Markdown / MDX 文件，Astro 在构建期生成静态页面，GitHub Actions 再把产物发布到 GitHub Pages。

## 为什么这样搭

这个博客要解决的是“长期沉淀”，不是一次性展示。它主要承载 AI Agent、影刀 RPA、Skill、数据处理和项目复盘类内容。

## 内容模型

每篇文章开头的 frontmatter 是内容进入站点的入口。它不只是元信息，也决定文章是否公开、如何聚合、是否进入精选。

  chart={`flowchart LR
    A[frontmatter] --> B[标题与摘要]
    A --> C[分类与标签]

固定分类现在只有四个：`AI Agent`、`影刀RPA`、`Skill`、`数据处理`。分类负责主线主题，标签负责细粒度索引。

## 写作

写作流程尽量短：先选主题和分类，再写 frontmatter，最后用 Markdown / MDX 组织正文。

## Agent 角色

Agent 不是发布系统本身，而是写作和维护流程里的协作者。它适合做结构整理、规则检查、图表补全和构建验证。

<VisualGrid columns={3}>
  <FeatureCard title="写作助手" meta="Draft" description="把草稿整理成视觉笔记，补齐标题、摘要、卡片和流程图。" />
  <FeatureCard title="规则审查" meta="Review" description="检查分类、草稿状态、敏感信息、frontmatter 和发布边界。" />
  <FeatureCard title="构建验证" meta="Verify" description="运行项目已有命令，确认文章能被 Astro 正常构建。" />
</VisualGrid>

<HighlightBox eyebrow="BOUNDARY" title="发布必须由人确认">
  Agent 可以准备草稿、重写文章和运行验证，但不能自行把新文章发布出去。是否从 draft: true 改成 draft: false，必须由用户明确确认。
</HighlightBox>

## 静态取舍

这个博客没有数据库、登录、评论和后台权限。它牺牲的是在线编辑便利性，换来的是低维护成本、可版本化内容和稳定访问。

  <FeatureCard title="静态博客" meta="当前选择" description="适合个人写作、版本管理、低成本部署和长期维护。">

</VisualGrid>
