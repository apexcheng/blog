# 代码 / 工程协作规则

本文件是本项目的代码 / 工程详细协作规则，适用于 Codex、Claude Code、OpenClaw 等代码 Agent。

核心目标：**最小改动、简单直接、可读可维护、不猜测、不做无关重构**。

---

## 入口关系

代码修改、构建配置、测试、依赖、工程结构或自动化脚本任务，先读根目录 `AGENTS.md`，再读本文件。

写作、审查、发布博客文章时，先读 `llms.txt` 和 `BLOG_AGENT.md`；不要从本文件展开文章规则。

---

## 0. 开发环境约束

本项目位于 WSL Linux 文件系统中，开发时必须优先使用 WSL 环境。

1. Codex、Claude Code、OpenClaw 等 Agent 执行项目命令时，应在 WSL 项目目录中运行。
2. 安装依赖、启动开发服务、运行构建、运行测试、执行 `npm` / `pnpm` / `node` 命令，都不要使用 Windows PowerShell、CMD 或 Windows 路径。
3. Windows 只作为浏览器预览环境使用，例如访问 `localhost`。
4. 不要把项目复制到 `/mnt/c`、桌面或其他 Windows 目录中开发。
5. 如果当前工具只能从 Windows shell 访问项目，必须明确说明“需切换到 WSL 环境运行验证”，不要把 Windows shell 中的结果当作最终验证。

### 0.1 WSL / Windows Shell 执行细节

部分 Agent 工具可能从 Windows PowerShell 发起命令，再转入 WSL 执行。为减少跨 shell 解析、路径和编码问题：

1. 项目命令优先统一通过 WSL bash 执行：
   ```bash
   wsl -d Ubuntu --cd /home/cheng/projects/personal-blog bash -lc '...'
   ```
2. 避免在 PowerShell 外层直接写复杂管道、`&&`、heredoc、多层引号、Markdown 反引号或大段中文文本。
3. 涉及中文、Markdown 反引号、复杂补丁、`git apply --cached`、行级暂存时，优先在 WSL 内处理；必要时在项目目录创建临时脚本，运行后立即删除，不纳入提交。
4. 不要从 PowerShell 管道直接向 WSL 传中文补丁或长文本，避免编码或转义被中途改写。
5. Node / npm 如果出现 WSL、Node 路径异常，先在 WSL bash 中确认并显式设置 PATH，例如：
   ```bash
   export PATH=/home/cheng/.local/bin:/home/cheng/.hermes/node/bin:$PATH
   ```
6. 不要把 Windows shell、路径转换或编码导致的失败直接判断为项目失败；先确认是否是跨 shell 执行问题。
7. 如果搜索、读取文件或带管道的命令被 PowerShell 把 `|`、引号或特殊字符拆开，默认先判断为跨 shell 转义 / 引号问题，不要直接推断为文件内容、命令目标或项目代码有问题；应改用更稳的方式在目标环境内重新执行并重新读取后，再继续判断。

## 1. 总体原则

1. 优先保证代码能运行，再考虑优化。
2. 优先简单、直接、容易审查的实现方式。
3. 不为“高级”“规范”“通用”而引入复杂设计。
4. 只解决当前任务明确要求的问题。
5. 不提前处理未出现、未验证、低概率的场景。
6. 不做与当前需求无关的重构、改名、格式化或结构调整。
7. 每一处改动都必须能对应当前需求。
8. 如果发现潜在问题，可以指出，但不要擅自扩大修改范围。
9. 不要编造不存在的功能、命令、目录、文件、XPath 或 API。
10. 不确定的地方必须标注“需运行验证”，不要自行推断。

---

## 2. 修改代码前流程

修改前先完成以下判断：

1. 当前问题是什么。
2. 证据来自哪里：报错、当前代码、文档、页面、用户描述。
3. 哪些文件 / 函数需要改。
4. 哪些内容不能改。
5. 最小可行修改是什么。
6. 修改后如何验证。

处理 Bug 时按以下顺序：

1. 先复现或定位问题。
2. 再找当前代码中的确定证据。
3. 只修改与问题直接相关的位置。
4. 修改后说明验证方式。
5. 不要把旧项目经验、未确认框架行为套进当前问题。

---

## 3. 代码风格

1. 代码要简单、直接、容易阅读。
2. 优先顺序式流程，不要为了少量逻辑拆很多函数。
3. 变量名直白，逻辑块清晰，数据流显式。
4. 临时变量可以短，但要直观，例如 `data`、`item`、`records`。
5. 不要使用 `x`、`tmp`、`obj` 这类过于抽象的变量名。
6. 保持现有项目风格，不要无关格式化整个文件。
7. 注释只解释关键业务逻辑、特殊页面逻辑或不直观的 API 行为。
8. 不要写解释 Python 基础语法的无意义注释。
9. 函数注释使用 Sphinx / reStructuredText 格式。

示例：

```python
def load_cache(file_path):
    """读取缓存文件。

    :param file_path: 缓存文件路径
    :type file_path: str
    :return: 缓存数据
    :rtype: dict
    """
    ...
```

## 4. 函数、常量、类的使用规则

### 4.1 函数

默认不要为了“看起来规范”拆函数。某段逻辑只在一个地方使用时，优先直接内联写在调用处，不要单独封装。

只有满足以下情况之一时，才抽出函数：

1. 代码量较多，内联后已经明显影响主流程阅读。
2. 逻辑本身较复杂，需要单独命名降低理解成本。
3. 后续已经明确会复用，而不是“可能会复用”。
4. 当前项目已有类似结构，需要保持风格一致。

以下情况默认不要抽函数：

1. 函数只在一个地方使用，且内联后主流程更直观。
2. 函数体只有 1 条语句。
3. 抽函数后反而需要更多参数传递，阅读更绕。
4. 单次调用、逻辑很短、且不代表明确业务步骤。

函数优先接收自己实际需要的具体值；如果只用 dict 里的少数字段，不要把整个 dict 传进去再拆。只有把 dict 当完整业务实体使用，或只是原样转发给下层时，才直接传容器。

### 4.2 常量

默认不要为了“看起来像常量”就提取常量。

只有满足以下情况之一时，才提常量：

1. 多处复用。
2. 属于公共配置。
3. 提取后明显更清楚。
4. 用户明确要求集中配置。

单次使用的 URL、XPath、字段名、固定字符串、小表达式，默认直接写在使用处。

展示值和业务值一致时，不要额外增加翻译函数、映射表或间接层。

### 4.3 类 / 管理器

默认不要新增类、管理器、抽象层。

只有满足以下情况之一时，才使用类：

1. 现有项目已经使用同类结构。
2. 状态复杂到函数无法清楚表达。
3. 用户明确要求长期架构设计。

### 4.4 配置层

默认不要新增复杂配置层。

但以下情况可以使用简单 `config.py`：

1. 用户明确要求使用 `config.py`。
2. 账号、路径、webhook、站点列表等需要集中维护。
3. 多平台、多环境配置明显会重复使用。

不要为了少量一次性参数设计复杂配置系统。
不要为了单次遍历构造中间分层结构；后续流程中不需要的结构不要提前设计。

## 5. 返回值解耦规则

1. 函数返回值优先只承载当前方法的主业务结果或不可逆的现场观测。
2. 不要为了后续计数、日志、汇总、通知等收尾步骤，顺手夹带额外统计信息。
3. 如果后置信息可以从主业务结果现场计算得到，就在最终使用处再算。
4. 只有当后置所需信息无法从主业务结果可靠推导，或现算会丢失必要上下文时，才额外返回辅助统计，并明确保留原因。

---

## 6. 输入、异常与兼容规则

1. 调用方已经明确约定入参类型或记录结构时，被调函数直接按约定使用。
2. 不要在业务逻辑中反复写 `isinstance`、`dict(...)`、空结构兜底等防御性归一化。
3. 如果输入形态确实不确定，只允许在边界统一归一化一次。
4. 进入业务逻辑后，不要继续重复判断和兜底。
5. 异常处理只覆盖当前业务明确需要处理的失败情况。
6. 不要默认添加复杂异常体系、重试框架、日志框架或配置层。
7. 捕获异常时必须保留关键错误信息。
8. 不要吞异常不说明。
9. 不要主动为未知旧接口或旧缓存增加兼容。
10. 已有代码中已经存在的兼容逻辑，不要无关删除。
11. 数据写入类任务要让失败隔离粒度和落盘时机对齐：逐项捕获异常就逐项保存，整批统一处理才在批量末尾统一保存。

---

## 7. 浏览器与网页自动化规则

1. 能用元素、选择器、页面接口或按钮状态验证，就不要优先坐标点击。
2. 中文或长文本输入优先考虑剪贴板输入。
3. 页面字段、按钮含义、价格口径等业务规则必须以当前需求或已验证证据为准，不要自行推断。
4. XPath、CSS、页面接口、按钮状态未验证时，必须标注“需运行验证”。
5. 价格、字段、状态等页面数据优先使用 XPath、CSS、页面接口或明确规则取数，不要让 Agent 自由猜测页面含义。

---

## 8. Excel / 表格处理规则

1. 表格大量写入时，优先考虑二维数组批量写入。
2. 不要逐单元格大量循环写入，除非数据量很小或当前项目只能这样做。
3. 读取和写入范围必须明确。
4. 写入前是否清空原内容，必须以用户需求或当前项目规则为准。
5. 不要自行改变 sheet 名、字段名、列顺序。
6. 表格路径、sheet 名、日期规则不确定时，先标注“需运行验证”。

---

## 9. 禁止行为

禁止以下行为：

1. 无关重构。
2. 无关格式化整个文件。
3. 擅自更换实现路线。
4. 引入不必要依赖。
5. 引入复杂封装、复杂抽象层、复杂配置系统。
6. 为低概率场景写大量防御代码。
7. 隐藏副作用。
8. 吞异常不说明。
9. 基于旧项目经验做推测。
10. 修改与当前需求无关的代码。
11. 编造不存在的 API、命令、目录、文件、XPath。
12. 自行猜测页面字段、按钮含义、价格口径。
13. 提交账号、密码、token、webhook、敏感路径。
14. 把“需运行验证”的内容描述成“已经验证通过”。

---

## 10. 最终输出要求

每次完成代码修改后，最终回复需要包含：

1. 修改了哪些文件。
2. 每个文件改了什么。
3. 是否符合最小改动。
4. 如何验证。
5. 哪些内容仍需运行验证。
6. 是否有未完成事项。

如果没有实际运行测试，必须明确说明“未运行测试”。
不要把静态检查、代码阅读描述成真实运行验证通过。

## 11. Agent 工具与协作补充

以下规则适用于支持对应能力的 Agent；不支持时按当前工具环境选择等价做法。

1. 搜索文件名、代码和文本时，优先使用 `rg` / `ripgrep`；不可用时再使用其他搜索工具。
2. 修改已有文件前，先阅读相关上下文；不要只凭文件名或旧经验直接改。
3. 手工改文件时优先使用补丁或编辑工具做最小 diff，避免无必要地整文件重写。
4. 不要用脚本批量改无关文件；确需批量处理时，先确认匹配范围只包含本次任务需要修改的文件。
5. 工作区可能已有用户改动；不要回滚、覆盖或清理不是本次任务产生的改动。
6. 不要执行 `git reset --hard`、`git checkout --` 等会丢弃改动的命令，除非用户明确要求。
7. 运行验证命令后，最终回复要说明关键结果；不要假设用户能看到终端输出。
8. 用户要求 code review / 审查时，先列问题，按严重程度排序，并给出文件和行号；如果没有发现问题，要明确说明未发现问题和剩余测试风险。

## 12. 核心原则

```text
最小改动 | 结构简单 | 流程显式 | 命名直白 | 少工程化 | 易审查 | 易维护
```
