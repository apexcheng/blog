# 文章结构与写作技能

这一类主要来自 `claude-blog` 的 `blog-write`、`blog-rewrite`、`content-rules`，并按本项目 `BLOG_AGENT.md` 的个人博客定位、默认视觉笔记形态和默认专业文字风格做了轻量化。

## 0. 默认形态和文字风格

默认文章形态是视觉笔记：优先用结论、指标卡片、流程图、模块分栏、关键细节和检查清单组织内容。

默认文字风格是专业风格：结构清晰、信息准确、逻辑完整，适合技术文章、项目说明、架构说明、部署说明和开发记录。可以使用必要术语，但不要故意写复杂，也不要写成营销文或官方公告。

大众阅读风格只在用户明确说“写给普通人看”“口语一点”“不要太专业”“普通读者能看懂”等情况使用。如果 Agent 判断主题明显更适合大众阅读，先提醒用户确认文字风格。大众阅读风格只影响文字表达，不改变视觉笔记这一主体形式。

## 1. 答案优先

每个主要小节先给结论，再解释。

适合本博客的段落模板：

```markdown
## [短标题]

结论先行：[一句话说明这个模块的判断或推荐做法]。

原因是：...
例子是：...
边界是：...
```

如果主题涉及流程、架构、状态流转或工具选择，优先配一张 Mermaid 或 `DecisionFlow`，不要只写长段落。

## 2. 视觉笔记结构

视觉笔记是本博客的默认主体形式。技术流程、架构、对比和决策内容优先采用这个结构：

```text
一句话结论
指标概览
核心路径 / 架构图 / 决策树
模块分栏
关键细节
检查清单 / 结论
```

对应组件：

| 信息类型 | 推荐表达 |
| --- | --- |
| 场景、输入、输出、风险 | `VisualGrid + MetricCard` |
| 模块、角色、步骤、能力边界 | `FeatureCard` |
| 强结论、提醒、检查清单 | `HighlightBox` |
| 判断顺序、步骤流 | `DecisionFlow` 或 Mermaid |
| 复杂分支、系统关系 | Mermaid |

## 3. 文章类型候选

从 `claude-blog` 的模板体系里挑出适合本博客的类型：

| 类型 | 适合主题 | 推荐结构 |
| --- | --- | --- |
| 工作流说明 | Agent 协作、发布流程、RPA 流程 | 输入 -> 决策 -> 执行 -> 验证 |
| 技术说明页 | Astro、MDX、组件、部署 | 结论 -> 架构 -> 关键配置 -> 检查 |
| 案例复盘 | 某次自动化或数据处理 | 背景 -> 问题 -> 方案 -> 结果 -> 教训 |
| 对比决策 | 工具、方案、路径选择 | 评价维度 -> 对比表 -> 推荐路径 |
| 清单型笔记 | Prompt、规则、坑位 | 分类 -> 清单 -> 示例 -> 使用建议 |
| 教程型文章 | 可复制步骤 | 前提 -> 步骤 -> 验证 -> 常见错误 |

## 4. 标题和小标题

本博客不必追求营销标题，但标题要短、能扫读。

规则：

- 主标题优先 4 到 12 个字。
- 二级标题负责分区，不塞整句解释。
- 卡片标题像标签，正文补充解释。
- 技术文章可用少量问题式标题，但不必强制 60% 都是问题。

候选标题模式：

```text
Agent 写作入口
发布前检查表
RPA 选择器优先
缓存保存时机
MDX 图表结构
```

## 5. 段落和列表

从 content-rules 保留这些轻量标准：

- 一段只讲一个点。
- 段落尽量 2 到 3 句。
- 超过 5 行时，改成列表、表格、卡片或图。
- 有 3 个以上并列项时用列表。
- 不连续堆列表，关键判断仍用短段落连接。

## 6. 图表节奏

本博客 `BLOG_AGENT.md` 已要求“至少考虑是否需要一张图”。可以进一步变成判断表：

| 内容 | 是否需要图 | 推荐图 |
| --- | --- | --- |
| 工具选择 | 需要 | 决策树 |
| 数据流 | 需要 | flowchart |
| 多角色协作 | 需要 | sequenceDiagram |
| 组件用法 | 可选 | VisualGrid |
| 单一观点短文 | 不强求 | HighlightBox 即可 |

## 7. 重写旧文的顺序

来自 `blog-rewrite`，但简化为最小改动版：

1. 先读原文，标出要保留的观点和个人经验。
2. 检查 frontmatter，不擅自发布。
3. 修结构：结论、分区、图表、清单。
4. 修事实：明显错误轻提醒；无来源数字可改为引用、经验判断或 TODO。
5. 修表达：删流水账、长段、占位词、重复解释。
6. 最后只做必要的格式调整，不整篇重写用户原本语气。

## 8. 个人经验是优势

`claude-blog` 强调 E-E-A-T，本博客可以转成更自然的“经验信号”：

- 我在这个项目里实际怎么处理。
- 这个方案踩过什么坑。
- 哪些结果是运行验证过的。
- 哪些只是当前判断，后面可能会改。
- 为什么没有选择更复杂的方案。

这比空泛的“最佳实践”更有价值。

## 9. MDX 教学流程图案例

当文章需要解释抽象概念时，不一定只用 Mermaid。像“什么是上下文”“Agent 如何拿到工具结果”“AI 工作流如何流转”这类教学型内容，更适合做成 MDX + Astro 组件：

```text
文章正文负责讲概念
        ↓
MDX 引入视觉组件
        ↓
组件用卡片、箭头、时间线表达
        ↓
读者看到的是一张可搜索、可响应式的说明图
```

适用场景：

| 场景 | 推荐形式 |
| --- | --- |
| 简单线性流程 | Mermaid `flowchart` |
| 复杂系统关系 | Mermaid 架构图 |
| 教学型解释图 | 自定义 Astro 组件 |
| 一次性封面配图 | PNG / WebP 图片 |

### 示例：上下文流程图

组件建议放在：

```text
src/components/notes/ContextFlow.astro
```

文章里这样引用：

```mdx
import ContextFlow from '../../components/notes/ContextFlow.astro';

## 什么是上下文？

<ContextFlow />
```

组件结构参考：

```astro
---
---

<section class="context-flow">
  <header>
    <h2>什么是上下文？</h2>
    <p>
      每次问答 / 对话时，聊天工具会把之前的聊天记录一起发给大模型。
      大模型每次只是接收到「输入」，然后生成「输出」，它本质上没有长期记忆。
    </p>
  </header>

  <div class="pipeline">
    <span>用户输入</span>
    <span>→</span>
    <span>聊天工具拼接上下文</span>
    <span>→</span>
    <span>大模型</span>
    <span>→</span>
    <span>模型输出</span>
  </div>

  <ol class="steps">
    <li>用户问：什么是 LLM？</li>
    <li>带上上一轮问题和模型回答，再继续对话。</li>
    <li>继续追加新的用户问题，形成更长上下文。</li>
    <li>模型看到的是当前输入 + 被一起附带的历史上下文。</li>
  </ol>
</section>
```

写作规则：

- 先用正文讲清楚概念，再插入组件。
- 组件只表达一个概念，不把整篇文章塞进组件。
- 文案要短，卡片标题像标签，长解释放回正文。
- 组件第一次实现可以写死内容，不急着抽象成通用组件。
- 同类图多了以后，再考虑提取 `TimelineCard`、`FlowStep` 这类子组件。
