# 可复用 Prompt 库

这些 prompt 是从三个参考项目提炼后的轻量版，贴合本博客的写作规则。可以直接复制给 Agent 使用，也可以以后升级成 Skill。

## 1. 文章 Brief

```text
请为我准备一份博客文章 brief。

主题：[主题]
目标读者：[读者]
我想表达的结论：[结论或先留空]
可用素材：[链接、旧文、代码、运行结果、经验]

要求：
1. 先判断文章类型：视觉笔记、技术说明、工作流复盘、RPA 经验、数据处理案例或其他。
2. 输出一句话结论、目标读者、文字风格、素材清单、视觉笔记结构、需要确认的问题。
3. 不确定的事实标注“需验证”，不要编造。
4. 默认文章形态是视觉笔记；默认文字风格是专业风格。
5. 大众阅读风格只有我明确要求或确认后才使用，并且只影响文字表达，不改变视觉笔记主体形式。
6. 结构要贴合本博客：结论、指标概览、流程/图表、模块拆解、关键细节、检查清单。
```

## 2. 视觉笔记大纲

```text
请把下面主题整理成本博客的视觉笔记大纲。

主题：[主题]
核心结论：[结论]
素材：[素材]

输出格式：
1. 标题候选 3 个，短标题优先。
2. 开头 1 到 2 句结论。
3. 3 到 4 个 MetricCard 指标建议。
4. 一张 Mermaid 或 DecisionFlow 图的内容草案。
5. 2 到 4 个 FeatureCard 模块建议。
6. 关键细节段落清单。
7. 结尾检查清单或 HighlightBox 文案。

要求：不要写成长篇流水账；没有必要用组件的地方保留普通 Markdown。
```

## 3. 新文章起草

```text
请根据这份 brief 起草一篇 MDX 博客文章。

Brief：
[粘贴 brief]

项目规则：
- 使用 templates/post.mdx 的 frontmatter 字段。
- 新文章保持 draft: true。
- category 只能是 AI Agent、影刀RPA、Skill、数据处理、视觉实验室。
- 默认文章形态是视觉笔记：用结论、指标卡片、流程图、模块分栏、关键细节和检查清单组织内容。
- 默认文字风格是专业风格：结构清晰、信息准确、逻辑完整，适合技术文章、项目说明、架构说明、部署说明和开发记录。
- 大众阅读风格只有我明确要求或确认后才使用，并且只影响文字表达，不改变视觉笔记主体形式。
- 开头先给 1 到 2 句结论。
- 视觉组件和图表优先服务于指标、对比、流程、架构、决策或重点结论，不要为了装饰强行堆组件。
- 代码块必须标注语言。
- 不确定事实写 TODO 或“需验证”，不要编造。
- 保留我的观点和个人判断，不要过度审查或改成安全稿。
- 不擅自发布，不修改业务代码。
```

## 4. 旧文重写

```text
请重写这篇旧文，但保持最小改动和原有观点。

文章路径：[路径]
重写目标：[例如结构更清楚 / 加图表 / 补事实来源 / 改成视觉笔记]

要求：
1. 先读原文，列出要保留的观点、经验和链接。
2. 给出改写计划，说明哪些地方会改，哪些不会改。
3. 不擅自改 draft 状态。
4. 优先修结构、事实、段落和占位词，不做无关重构。
5. 保留作者观点和个人味道；明显错误轻提醒，不要整篇改成安全稿。
6. 修改后说明如何验证；未运行测试必须明确说未运行。
```

## 5. 事实核查

```text
请对这篇文章做事实核查：[路径]

检查范围：
- 数字、百分比、年份、版本、价格。
- 带来源的强断言。
- 没有来源但听起来像事实的句子。

输出：
| 位置 | 说法 | 状态 | 建议 |

状态只能用：已验证、部分支持、未找到、无来源、需人工验证。

要求：不要把读过原文当成验证；只有打开来源并确认支持时才写已验证。
补充：事实核查只处理事实证据，不接管作者观点；明显错误可以建议修改，不要把文章改成没有个人味道的安全稿。
```

## 6. 发布前审稿

```text
请按本项目 BLOG_AGENT.md 对这篇文章做发布前审稿：[路径]

优先检查：
1. 是否有敏感信息、私密路径、token、账号。
2. 是否有模板占位词。
3. 结构是否清楚，是否需要图表或分栏。
4. Mermaid 图是否只表达一个问题，节点是否短。
5. public/files 引用是否可公开。
6. 是否有空泛开头、抽象大词或没有例子的“很重要”。
7. 是否保持视觉笔记主体形态；大众阅读风格只影响文字表达，默认文字风格应为专业风格，除非用户明确要求或确认。

输出：直接列出发现的问题和建议；没有问题就说明未发现明显问题。最后说明未运行或已运行的验证。
```

## 7. AI 味清理

```text
请帮我清理这段文字的 AI 味，但不要改掉原意。

文本：
[粘贴文本]

要求：
- 删除空泛开头和套话。
- 用具体场景、动作、文件、流程替代抽象大词。
- 保留作者自己的判断和语气。
- 不要为了“更正式”而加长句子。
- 输出改写后文本，并列出 3 条你改动的原则。
```

## 8. 作者 Voice 提炼

```text
请从下面 1 到 3 篇旧文中提炼我的博客写作 voice。

文章：
[路径或正文]

输出：
1. 常见开头方式。
2. 句子和段落长度偏好。
3. 喜欢的结构。
4. 常用词和避免词。
5. 适合写进 BLOG_AGENT.md 的 5 到 8 条规则。

要求：只基于样本，不要脑补品牌人格。
```

## 9. 内容索引维护

```text
请根据这篇新文章更新 CONTENT_INDEX.md。

文章路径：[路径]

要求：
- 只新增或更新对应一行。
- 不改变表头和列顺序。
- 分类使用文章 frontmatter 的 category。
- 主题概览用一句短话。
- “适合什么时候参考”写成具体使用场景。
```

## 10. Skill 化评估

```text
请评估下面这个博客工作流是否值得做成 Skill。

工作流：[描述]

请判断：
1. 是否重复出现。
2. 输入输出是否稳定。
3. 是否需要按需读取 reference。
4. 是否有可验证成功标准。
5. 更适合写进 BLOG_AGENT.md、prompt 库，还是单独 Skill。

最后给出一个最小 Skill 草案，包括 name、description、流程和输出格式。
```
