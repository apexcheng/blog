# 审稿与发布技能

这一类主要来自 `claude-blog` 的 `blog-reviewer`、`blog-seo-check`、`blog-factcheck`，并按本项目发布规则简化。

## 1. 本项目发布前检查

保留轻量检查，避免把审稿做成复杂评分系统：

- 没有账号、token、公司内部资料、私密路径。
- 没有“待填写”“replace me”“模块一”等模板残留。
- 默认保持视觉笔记主体形态，技术流程、架构或决策类内容优先考虑图表或分栏。
- 引用 `public/files/` 时确认文件可公开。
- 文字风格符合用户要求；默认专业风格，大众阅读风格需要用户明确要求或确认，且不改变视觉笔记主体形态。
- 明显事实错误可以提醒；不要把审稿变成全面纠错、政治规避或观点重写。

## 2. 可读性和内部链接检查

本博客不是 SEO 站，保留低成本的可读性检查：

- 标题是否具体，不泛泛而谈。
- 描述是否说明读者能得到什么。
- 内部是否链接到 1 到 3 篇相关旧文。
- 图片或图表是否有简短说明。
- 代码块是否标注语言。
- Mermaid 图是否只表达一个问题。

不建议强制：

- 每篇 150 到 160 字 meta description。
- 每篇固定数量外链、统计数据、FAQ schema。
- 强制问题式 H2 比例。

## 3. 事实核查报告格式

只有用户明确要求事实核查，或文章包含明显需要验证的数字、版本、价格、法律政策等内容时，才让 Agent 输出这种报告：

```markdown
## 事实核查报告：[文章]

| 位置 | 说法 | 状态 | 建议 |
| --- | --- | --- | --- |
| H2: 工具选择 | “X 支持 Y” | 需验证 | 补官方链接或改成经验描述 |
| 第 3 段 | “效率提升 50%” | 无来源 | 删除数字或补测试方法 |

## 必须处理
1. ...

## 可选优化
1. ...
```

## 4. AI 味和空话检查

从 `blog-reviewer` 保留一部分，但不要过度机械：

重点检查：

- 开头是否空泛，例如“在当今快速发展的时代”。
- 是否大量使用“赋能、革新、无缝、深入探讨、复杂生态”等虚词。
- 是否每段长度和句式都很平均。
- 是否用很多看似平衡但没结论的句子。
- 是否只说“很重要”，没有例子、证据或操作。

改法：

- 用具体场景替代大词。
- 用项目里的真实文件、命令、流程替代抽象建议。
- 删掉不能服务结论的过渡句。
