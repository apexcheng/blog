# 开发规则

本文件是本项目的代码 / 工程协作规则。处理代码修改、样式调整、构建配置、测试、依赖、项目结构或 Git 操作时读取。

## 核心原则

```text
最小改动 | 结构简单 | 只改相关文件 | 保持现有风格 | 不做无关重构
```

## 开发环境

1. 项目位于 WSL Linux 文件系统中，项目命令优先在 WSL 项目目录运行。
2. Windows 主要作为浏览器预览环境使用，不要把 Windows 路径问题直接判断为项目失败。
3. 运行 `npm`、构建、测试、搜索文件时，优先在项目根目录执行。
4. Node / npm / pnpm 验证优先在 WSL 内执行；如果命令解析到 Windows Node、WSL 版本不兼容或 `node_modules` 可选依赖异常，先按执行环境问题排查，不要直接判断为项目构建失败。

## 修改边界

1. 修改前先阅读相关文件，不凭文件名或旧经验直接改。
2. 只修改当前任务直接相关的文件和内容。
3. 不做无关格式化、无关改名、无关重构。
4. 不引入不必要依赖。
5. 工作区可能已有用户改动；不要回滚、覆盖或清理不是本次任务产生的改动。
6. 发现无关问题可以说明，但不要擅自扩大修改范围。

## 代码风格

1. 优先简单、直接、可读。
2. 保持现有项目风格。
3. 不为一次性逻辑增加复杂抽象。
4. 函数、常量、配置层只在明显复用、明显降低理解成本或用户明确要求时新增。
5. 注释只解释关键业务逻辑、特殊页面逻辑或不直观的行为。

## 文章与代码边界

1. 正式文章目录是 `src/content/posts/`。
2. 视觉实验和 Demo 文章目录是 `src/content/posts/visual-lab/`。
3. Agent 规则和索引放在 `docs/`，不要写入正式文章目录。
4. 写文章、改文章、发布文章时，按 `docs/writing/` 下的规则执行。

## 验证要求

1. 修改代码、样式、构建配置后，优先运行相关测试或构建。
2. 只改文档时，可以不运行构建，但最终回复要说明未运行测试。
3. 不确定的事实、页面行为、工具参数或构建结果必须标注“需运行验证”。
4. 最终回复说明修改文件、修改内容、验证方式和未完成事项。

## 规则维护边界

1. 更新规则时，先判断规则应该放在哪个现有文件，不要默认新建目录或新建规则文件。
2. `AGENTS.md` 只放硬规则和路由；详细开发规则放在 `docs/DEV_RULES.md`。
3. 写作规则放在 `docs/writing/`，不要放回根目录。
4. 不恢复旧的 `docs/agent/` 结构，除非用户明确要求。
5. 同一条规则只保留一个权威版本，其他文件只做简短引用。
6. 新增规则必须能对应长期维护价值，不为单次问题堆叠长期规则。

## Git 约定

1. 本项目默认使用 `main` 分支。
2. 不新建长期分支，除非用户明确要求。
3. 提交或推送前确认目标分支是 `main`。
