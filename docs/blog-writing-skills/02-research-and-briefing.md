# 研究与 Brief 技能

这一类主要来自 `claude-blog` 的 `blog-brief`、`blog-outline`、`blog-strategy`、`blog-discourse`、`blog-factcheck` 和研究质量 reference。适合改造成“写文章前的准备动作”。

## 1. 文章 Brief

用途：在动笔前把文章目的、读者、素材、结构和风险定下来。

适合本博客的轻量输出：

```text
# 文章 Brief：[主题]

## 一句话结论
[本文最想让读者带走的判断]

## 目标读者
- 读者是谁
- 已经知道什么
- 看完后能做什么

## 文章类型
[视觉笔记 / 技术说明 / 工作流复盘 / RPA 经验 / 数据处理案例]

## 素材清单
- 已有事实
- 可引用旧文
- 需要外部资料
- 需要用户确认

## 推荐结构
1. 结论
2. 指标概览
3. 流程或决策图
4. 模块拆解
5. 关键细节
6. 检查清单

## 风险
- 事实不确定
- 隐私或敏感信息
- 需要运行验证
```

## 2. 研究质量 Rubric

从 `claude-blog` 的 research-quality 提炼，适合每次外部资料整理后自查。

| 维度 | 问题 | 本博客用法 |
| --- | --- | --- |
| Groundedness | 关键说法有没有来源 | 事实、数字、工具行为必须有来源或标注需验证 |
| Specificity | 是否具体 | 用具体工具、版本、页面、字段、例子替代泛泛说法 |
| Coverage | 是否单一来源 | 重要判断至少交叉看两类来源：官方文档 + 实践讨论 |
| Actionability | 读者能否照做 | 写清输入、步骤、输出、失败处理 |
| Format compliance | 格式是否干净 | 内联链接、无重复来源块、无占位词 |

轻量通过线：总分不必机械化，但如果有“关键事实无来源”或“读者看完不知道怎么做”，不要进入正文。

## 3. 选题预检

有些题目直接搜索会得到垃圾信息。写前先判断：

| 陷阱 | 例子 | 改法 |
| --- | --- | --- |
| 太泛 | “Agent” | 改成“Agent 写博客的审稿流程” |
| 太像教程标题 | “怎么用 Docker” | 改成“Docker 本地开发循环里最容易出错的 3 个点” |
| 数字干扰 | “40 岁的人适合什么工具” | 去掉无意义数字，改成场景或角色 |
| 无读者 | “Excel 自动化” | 补充读者：运营、财务、RPA 维护者 |

## 4. Named Entity Decomposition

写产品、工具、框架或具体项目时，不要只搜一个词。先拆成查询面：

```text
主题：Codex Skill 写作流程

1. 官方定义：Agent Skills specification / official docs
2. 实现示例：anthropics/skills
3. 真实生态：awesome-agent-skills 中同类技能
4. 垂直案例：claude-blog 的写作技能链
5. 本博客适配：BLOG_AGENT.md、已有 Skill 类文章
```

## 5. 权威资料 + 实践讨论双轨

`blog-discourse` 的启发是：权威资料回答“正确是什么”，实践讨论回答“真实使用中大家卡在哪里”。

本博客可以简化成两栏：

| 资料类型 | 适合回答 | 来源例子 |
| --- | --- | --- |
| 权威资料 | API、规则、版本、官方建议 | 官方文档、README、规范、源码 |
| 实践讨论 | 坑、误区、取舍、真实反馈 | GitHub issue、论坛、博客、评论 |

输出时明确区分：

```text
官方规则：...
实践观察：...
我的建议：...
仍需验证：...
```

## 6. 事实核查清单

来自 `blog-factcheck`，改成发布前手工检查：

- 所有数字、百分比、年份、价格、版本号是否有来源。
- 引用链接是否能打开。
- 来源页面是否真的支持该结论。
- PDF、登录页、动态页面是否需要人工验证。
- 没有来源的强断言是否改成经验判断或 TODO。
- 旧资料是否只是历史背景，而不是当前结论依据。

## 7. 研究输出不要写成资料堆

`synthesis-contract` 最值得保留的规则：

- 每段只表达一个离散 claim。
- 引用用内联 Markdown 链接。
- 不要在结尾重复堆一个 Sources 列表。
- 不要编造没见过的标题。
- 不要把搜索结果排名、分数、原始聚类直接贴进正文。

适合本博客的合成句式：

```text
[来源] 给出的规则是 X；[实践来源] 暴露的问题是 Y。因此本文采用 Z，代价是 W。
```
