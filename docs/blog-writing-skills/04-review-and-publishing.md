# 审稿与发布技能

这一类主要来自 `claude-blog` 的 `blog-reviewer`、`blog-seo-check`、`blog-factcheck`，并按本项目发布规则简化。

## 1. 审稿优先级

审稿时先找会造成真实问题的内容，再看优化项。

| 优先级 | 问题 | 处理 |
| --- | --- | --- |
| P0 | 敏感信息、token、账号、私密路径 | 必须删除或脱敏 |
| P0 | 未确认发布却 `draft: false` | 改回 `draft: true` 或问用户 |
| P0 | 明显编造事实、数字、来源 | 删除、改写或标 TODO |
| P1 | frontmatter 缺字段或分类错误 | 按模板修正 |
| P1 | 残留占位词 | 删除或补全 |
| P1 | 结构混乱，读者不知道结论 | 重排结构 |
| P2 | 图表可读性、标题过长、段落过长 | 适度优化 |
| P3 | SEO、社交摘要、可分发性 | 可选优化 |

## 2. 本项目发布前检查

直接继承 `BLOG_AGENT.md` 的规则：

- frontmatter 字段完整。
- `category` 只能是 `AI Agent`、`影刀RPA`、`Skill`、`数据处理`。
- 新文章默认 `draft: true`。
- 只有用户明确确认发布，才改 `draft: false`。
- 没有账号、token、公司内部资料、私密路径。
- 没有“待填写”“replace me”“模块一”等模板残留。
- 技术流程、架构或决策类内容优先考虑图表或分栏。
- 引用 `public/files/` 时确认文件可公开。

## 3. 轻量 SEO 检查

本博客不是 SEO 站，但这些检查成本低：

- 标题是否具体，不泛泛而谈。
- 描述是否说明读者能得到什么。
- URL slug 是否短、可读、英文小写连字符。
- 内部是否链接到 1 到 3 篇相关旧文。
- 图片或图表是否有简短说明。
- 代码块是否标注语言。
- Mermaid 图是否只表达一个问题。

不建议强制：

- 每篇 150 到 160 字 meta description。
- 每篇固定数量外链、统计数据、FAQ schema。
- 强制问题式 H2 比例。

## 4. 事实核查报告格式

可让 Agent 输出这种审稿报告：

```markdown
## 事实核查报告：[文章]

| 位置 | 说法 | 状态 | 建议 |
| --- | --- | --- | --- |
| H2: 工具选择 | “X 支持 Y” | 需验证 | 补官方链接或改成经验描述 |
| 第 3 段 | “效率提升 50%” | 无来源 | 删除数字或补测试方法 |

## 必须处理
1. ...

## 可选优化
1. ...
```

## 5. AI 味和空话检查

从 `blog-reviewer` 保留一部分，但不要过度机械：

重点检查：

- 开头是否空泛，例如“在当今快速发展的时代”。
- 是否大量使用“赋能、革新、无缝、深入探讨、复杂生态”等虚词。
- 是否每段长度和句式都很平均。
- 是否用很多看似平衡但没结论的句子。
- 是否只说“很重要”，没有例子、证据或操作。

改法：

- 用具体场景替代大词。
- 用项目里的真实文件、命令、流程替代抽象建议。
- 删掉不能服务结论的过渡句。

## 6. Delivery Contract 的轻量版

`claude-blog` 有 5-gate delivery contract，本博客可以先用 4 个门：

```text
Gate 1: 格式
- frontmatter 正确
- draft 状态正确
- 分类正确

Gate 2: 内容
- 开头有结论
- 结构可扫读
- 没有占位词

Gate 3: 事实与隐私
- 数字和强断言有来源或标注
- 无敏感信息
- public/files 引用可公开

Gate 4: 运行验证
- 修改文章后至少运行可用的测试或构建
- 如果未运行，最终说明“未运行测试”
```

## 7. 审稿输出要按严重程度排序

如果用户说“审查 / review”，输出先列问题：

```markdown
## 问题

1. [P0] 文件:行号 - 问题
   影响：...
   建议：...

## 剩余风险
- ...

## 简短总结
- ...
```

没有问题时也要明确说“未发现阻塞问题”，并说明哪些测试或人工验证还没做。
